# RustChain Node API Documentation

## Overview

This documentation provides comprehensive API reference for the RustChain node. The OpenAPI 3.0 specification is available in `openapi_spec.yaml` and can be viewed interactively using the Swagger UI at `swagger-ui.html`.

## API Endpoints

### Public Endpoints (No Authentication Required)

| Endpoint | Method | Description | Response |
|----------|--------|-------------|----------|
| `/health` | GET | Node health check | 200 OK with health status |
| `/ready` | GET | Readiness probe for k8s | 200 OK when ready |
| `/api/miners` | GET | List of active miners | JSON array of miner objects |
| `/api/epoch` | GET | Current epoch information | JSON with epoch details |
| `/api/ledger` | GET | Rewards ledger | JSON with reward history |

### Authenticated Endpoints

| Endpoint | Method | Description | Authentication |
|----------|--------|-------------|----------------|
| `/attest/submit` | POST | Submit hardware attestation | Hardware fingerprint validation |
| `/api/balance` | GET | Query RTC balance | `miner_id` parameter required |
| `/api/transfer` | POST | Signed RTC transfer | Cryptographic signature required |

```bash
curl https://50.28.86.131/api/miners
# Returns: [{"miner_id": "dual-g4-125", "arch": "G4", "last_attest": 1740783600}, ...]
```

### Query Balance
```bash
curl "https://50.28.86.131/api/balance?miner_id=dual-g4-125"
# Returns: {"miner_id": "dual-g4-125", "amount_rtc": 150.0}
```

## Error Handling

All API endpoints follow standard HTTP status codes:

- **200 OK**: Successful request
- **400 Bad Request**: Invalid parameters or malformed request
- **401 Unauthorized**: Authentication required or failed
- **404 Not Found**: Endpoint or resource not found
- **500 Internal Server Error**: Server-side error

## Rate Limiting

The API implements rate limiting to prevent abuse:

- **Public endpoints**: 100 requests per minute per IP
- **Authenticated endpoints**: 50 requests per minute per miner_id
- **Attestation endpoint**: 1 request per epoch per miner_id

## Versioning

API versioning follows semantic versioning:

- **Major version changes**: Breaking changes to API structure
- **Minor version changes**: Backward-compatible additions
- **Patch version changes**: Bug fixes and internal improvements
